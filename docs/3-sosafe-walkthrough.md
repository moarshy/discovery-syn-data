# SoSafe Synthetic Data Pipeline — Team Walkthrough

This document walks through the full data generation and analysis pipeline for SoSafe, step by step. Each stage shows what was generated, how to inspect it, and what patterns emerge from the data.

**Pipeline**: Structured data → Unstructured data (Claude) → Graph → Vector index → Evaluation

**To reproduce**: Run from `round2/synthetic-data-gen/`:
```bash
uv sync
export $(grep ANTHROPIC_API_KEY ../../.env | xargs)
uv run python -m src.generate_sosafe
uv run python -m src.generate_unstructured sosafe
uv run python -m src.build_graph sosafe
uv run python -m src.naive_rag sosafe
uv run python -m src.evaluate sosafe
```

**To browse the database**: `uv run datasette data/sosafe.db -p 8787` → open http://localhost:8787

## Stage 1: Structured Data Generation (~10 seconds)

`python -m src.generate_sosafe` creates `data/sosafe.db` with 4 tables.

### How it's generated

No LLM calls here — everything is algorithmic, deterministic (seeded), and fast.

**Config-driven**: A `PLAN_TIERS` dict defines the shape of each tier (Essential / Professional / Premium / Ultimate) with: distribution ratio, ARR range, churn rate, login frequency, campaign frequency, and which features the tier is entitled to. A `FEATURES` list defines 8 features with activation difficulty (low / medium / high). These two configs drive everything downstream.

**Libraries**: Faker for company names and dates, NumPy for distributions (uniform for ARR, lognormal for employee count, weighted random for tier assignment), Python random for coin-flip decisions.

**Key behavioral rules baked into the generator**:
- Churned accounts get **0.5x activation probability** — they were less likely to turn on features
- Churned accounts get **0.6x session count** and **skip ~60% of sessions** in the 30-60 days before their churn date — simulating the activity drop-off before leaving
- Campaign frequency **scales with tier** — Ultimate runs 4x more campaigns than Essential
- Feature activation dates follow an **exponential distribution** (mean 30 days) from contract start — most activations happen early

**State machines for events**: Each account's activity is generated by running state machines that model real platform behavior:
- **Admin session**: `login → dashboard_viewed → [30% chance: campaign_created → [70% chance: campaign_launched]]`
- **Phishing simulation**: `email_sent(N) → email_opened(40-80%) → link_clicked(5-25%) → [30% chance: data_entered(10-40%)] → email_reported(2-15%)`
- **E-learning**: `module_assigned(N) → module_started(50-90%) → module_completed(40-85%)`
- **Feature activation**: one event per activated feature, timestamped to the activation date

### 1.1 Accounts (200 rows)

Each row is a B2B customer with plan, revenue, and churn status.

| Plan Tier | Accounts | Avg ARR | Avg Employees | Churn Rate |
|-----------|----------|---------|---------------|------------|
| Essential | 70 | $12,277 | 1,418 | 29% |
| Professional | 66 | $49,814 | 5,875 | 11% |
| Premium | 40 | $160,816 | 19,904 | 5% |
| Ultimate | 24 | $558,897 | 59,600 | 8% |

**Observation**: Churn rate drops 5x from Essential (29%) to Premium (5%). Ultimate is slightly higher at 8% — these are large accounts where a single champion departure can trigger churn despite high ARR.

```sql

FROM accounts GROUP BY plan_tier ORDER BY avg_arr;
```

## Stage 2: Unstructured Data Generation (~6 minutes with 10 parallel workers)

`python -m src.generate_unstructured sosafe` adds 4 tables to `sosafe.db`. Each record is generated by Claude, conditioned on the account's structured data (plan, features, churn status).

### How it's generated

Every piece of unstructured data is produced by Claude Sonnet, called via the Anthropic async API with 10 concurrent workers and retry-with-backoff on rate limits. Total: ~70 API calls.

**The key design principle**: each Claude call receives the account's *actual structured data* as context — plan tier, ARR, activated features, unactivated features, churn status. The prompt then tells Claude to generate text that is *consistent with those signals*. This means a churned Essential account with no features activated will get frustrated tickets and negative reviews, while a retained Ultimate account with full activation will get positive reviews and routine QBR transcripts.

**System prompt** (same for all 4 data types):
> *"You are a synthetic data generator for a B2B SaaS cybersecurity awareness platform (similar to SoSafe). Generate realistic business data that reflects real-world patterns. Return ONLY valid JSON with no markdown fencing or commentary."*

**Account context** (passed to every prompt as JSON):
```json
{
  "account_id": "ACC-0042",
  "name": "Martinez LLC",
  "plan_tier": "Essential",
  "industry": "Healthcare",
  "employees": 892,
  "arr": 8234,
  "churned": true,
  "activated_features": ["basic_phishing"],
  "unactivated_features": ["elearning"]
}
```

Below is the gist of each prompt and how it steers Claude's output.

### 2.2 Transcripts (50 rows)

**Generation**: 50 individual Claude calls (one per transcript), 10 concurrent. The type distribution is fixed: 40% onboarding, 25% QBR, 20% escalation, 15% churn exit. Churn exit transcripts are always generated for churned accounts.

**Prompt gist** (example for a churn exit call):
> Generate a realistic churn exit call transcript for a B2B cybersecurity awareness platform.
> Account context: `{...account JSON with plan, features, churn status...}`
>
> Transcript type: churn_exit
> - This is a churn exit interview. The customer is leaving. Explore why.
>
> Format as a realistic dialogue between CSM (Customer Success Manager) and the customer contact.
> Include 8-15 exchanges. Reference specific features and metrics from the context.
>
> Return JSON: `{account_id, type, date, content, summary}`

The guidance line changes per type: onboarding → "Walk through initial setup and feature activation", QBR → "Discuss usage metrics, feature adoption, ROI", escalation → "The customer has an urgent issue."

| Type | Count | Purpose |
|------|-------|---------|
| Onboarding | 20 | Initial setup, feature walkthrough |

**Observation**: 93% of churned account reviews are negative. Zero positive reviews from churned accounts. Retained accounts have a healthy 61% neutral / 34% positive split — the 8% negative from retained accounts are worth investigating as early warning signals.

### 2.4 Churn Reasons (31 rows — one per churned account)

**Generation**: All churned accounts batched in groups of 10, 10 concurrent. Unlike the others, this isn't sampled — every churned account gets exactly one churn reason.

**Prompt gist**:
> For each of these churned B2B SaaS accounts, generate a realistic churn reason tied to their actual data signals.
> Account contexts: `[...churned account JSON objects...]`
>
> Churn reason categories: "poor_adoption", "budget_cuts", "competitor_switch", "missing_features", "poor_support", "champion_left", "low_roi"
>

> - Many unactivated features → "poor_adoption" or "low_roi"

**Why this matters**: The correlation rules in the prompt mean Claude picks a reason category that makes sense given the account's data profile. An Essential account with $8K ARR and 1 activated feature will get "poor_adoption" or "budget_cuts", not "champion_left". This internal consistency is what makes the downstream analysis chain work.

## Stage 3: Graph Construction (~5 seconds)

`python -m src.build_graph sosafe` builds a NetworkX directed graph from all 8 SQLite tables.

### How it's generated

No LLM calls. Reads all 8 SQLite tables and converts them into a NetworkX DiGraph by iterating rows and creating nodes and edges:

1. **Account nodes** (200): one per account, with all profile attributes stored on the node
2. **Feature nodes** (8): one per feature, shared across accounts
3. **Campaign nodes** (4,062): one per campaign
4. **Event nodes** (61,194): one per event
5. **Ticket nodes** (500): one per support ticket
6. **Edges**: `Account→Feature` (ENTITLED_TO/ACTIVATED), `Account→Campaign` (RAN), `Account→Event` (GENERATED), `Campaign→Event` (PRODUCED), `Account→Ticket` (FILED), `Ticket→Feature` (RELATES_TO via a category-to-feature lookup map)
7. **Temporal chains**: Events are grouped by account_id, sorted by timestamp, and linked with NEXT edges — creating a per-account timeline that retrieval functions can walk

### Graph Stats

| Metric | Value |
|--------|-------|
| Total nodes | 65,964 |
| Total edges | 142,906 |

| Node Type | Count | What It Represents |
|-----------|-------|--------------------|
| Account | 200 | Customer organizations |
| Feature | 8 | Product capabilities |
| Campaign | 4,062 | Phishing sims + e-learning |
| Event | 61,194 | Activity events (temporal) |
| Ticket | 500 | Support tickets |

| Edge Type | Count | What It Connects |
|-----------|-------|--------------------|
| GENERATED | 61,194 | Account → Event |
| NEXT | 60,994 | Event → Event (temporal chain) |
| PRODUCED | 14,758 | Campaign → Event |
| RAN | 4,062 | Account → Campaign |
| RELATES_TO | 562 | Ticket → Feature |
| FILED | 500 | Account → Ticket |
| ACTIVATED | 496 | Account → Feature (turned on) |
| ENTITLED_TO | 340 | Account → Feature (has access) |

**Observation**: The gap between ENTITLED_TO (340 edges) and ACTIVATED (496 edges) seems inverted — but remember, accounts can have both an ENTITLED_TO and an ACTIVATED edge to the same feature. The real signal is the difference: 340 entitlements that are NOT activated. That's the product gap.

#### How to explore the graph

```python
from src.build_graph import load_graph, print_graph_stats

G = load_graph("graph/sosafe_graph.pkl")
print_graph_stats(G)

# Inspect one account's neighborhood
acct = "account_ACC-0001"
print(G.nodes[acct])  # Node attributes

# What features is this account entitled to vs activated?
for _, target, data in G.edges(acct, data=True):
    if data["edge_type"] in ("ENTITLED_TO", "ACTIVATED"):
        feat = G.nodes[target]["feature_name"]
        print(f"  {data['edge_type']} → {feat}")

# How many events does this account have?
events = [t for _, t, d in G.edges(acct, data=True) if d["edge_type"] == "GENERATED"]
print(f"  {len(events)} events")

# How many tickets?
tickets = [t for _, t, d in G.edges(acct, data=True) if d["edge_type"] == "FILED"]
print(f"  {len(tickets)} tickets")
```

#### Why the graph matters

The graph enables queries that SQLite joins can't easily answer:
- **Temporal chains**: Follow NEXT edges to see event sequences per account — what happened in what order
- **Multi-hop traversal**: Account → Ticket → Feature (via RELATES_TO) — which features generate the most support friction?
- **Pattern aggregation**: Walk all churned accounts' event chains, extract common patterns, compare to retained accounts
- **Neighborhood comparison**: Compare the full subgraph structure of a churned vs retained account

## Stage 4: Vector Index (~3 seconds)

`python -m src.naive_rag sosafe` creates a FAISS index with 200 documents (one per account).

### How it's generated

No LLM calls. For each of the 200 accounts, `account_to_document()` reads the account's structured data from SQLite (profile, activations, campaigns, events, tickets) and concatenates it into a single text string. These 200 strings are then encoded into 384-dimensional vectors using `sentence-transformers` (all-MiniLM-L6-v2), L2-normalized, and inserted into a FAISS `IndexFlatIP` index for cosine similarity search.

Each document is a text summary of the account:
```
Account ACC-0001 (Buchanan and Sons) | Plan: Essential | ARR: $15,234 |
Industry: Telecom | Employees: 4,062 | Churned: True |
Activated features: basic_phishing |
Entitled but NOT activated: elearning |
Campaigns: 3 phishing sims, 2 e-learning |
Events (245 total): admin.login(82), admin.dashboard_viewed(82), ... |
Tickets (3 total): Billing/Licensing(2), Integration Setup(1)
```

This is the **baseline** for comparison. When you ask a question, it finds the 10 most similar account documents by cosine similarity and passes them to Claude. The limitation: it can only surface individual accounts that match the query text — it can't aggregate, compare cohorts, or trace temporal patterns.

## Stage 5: Evaluation (~3 minutes)

`python -m src.evaluate sosafe` runs 5 questions through both GraphRAG and naive RAG, then compares the responses.

### How the evaluation works

Each question goes through two parallel pipelines that differ **only in how they retrieve context** — the LLM, prompt structure, and scoring are identical.

#### Pipeline A: GraphRAG

```
Question
  → Retrieval function walks the NetworkX graph
    (e.g., find_activation_gaps() iterates all 200 Account nodes,
     follows ENTITLED_TO and ACTIVATED edges to Feature nodes,
     groups by plan_tier, computes activation rates)
  → Serialization function converts the graph traversal results
    into a structured text block (markdown tables, bullet lists)
  → Claude receives: system prompt + serialized context + question
  → Response
```

Each of the 5 questions has a dedicated retrieval function in `src/retrieval.py` that knows *which part of the graph to walk*:

| Question | Retrieval function | What it traverses |
|----------|-------------------|-------------------|
| Q1 (churn ↔ activation) | `query_churn_activation()` | All churned Account nodes → their ACTIVATED/ENTITLED_TO edges → Feature nodes. Also runs `find_common_patterns()` to extract event sequences. |
| Q2 (pre-churn signals) | `query_pre_churn_behavior()` | Churned accounts → their GENERATED edges → Event nodes (sorted by timestamp). Counts unactivated features, last event types, ticket categories. |
| Q3 (tier comparison) | `query_tier_comparison()` | All Essential accounts vs all Premium accounts. For each, counts events, campaigns, activated features, tickets, churn rate. |
| Q4 (activation gaps) | `query_activation_gaps()` | Every Account node → every Feature node edge. Groups by tier, computes entitled vs activated counts per feature. |
| Q5 (ticket ↔ churn) | `query_ticket_correlation()` | All Account nodes → FILED edges → Ticket nodes. Splits by churned/retained, counts categories and priorities. |

The `find_activation_gaps()` function walks the graph and `serialize_gaps_for_llm()` produces text like:

#### Pipeline B: Naive RAG

```
Question
  → sentence-transformers encodes the question into a 384-dim vector
  → FAISS cosine similarity search finds the 10 closest account documents
  → The 10 document texts are concatenated into a context block
  → Claude receives: system prompt + top-10 documents + question
  → Response
```

Each account document is a single text string built by `account_to_document()`:

```
Account ACC-0042 (Martinez LLC) | Plan: Essential | ARR: $8,234 |
Industry: Healthcare | Employees: 892 | Country: Germany | Churned: True |
Activated features: basic_phishing |
Entitled but NOT activated: elearning |
Campaigns: 3 phishing sims, 1 e-learning |
Events (145 total): admin.login(48), admin.dashboard_viewed(48),
  admin.campaign_created(12), simulation.email_sent(8), ... |
Tickets (4 total): Billing/Licensing(2), Bug Report(1), Integration Setup(1)
```

```
... 8 more accounts ...
```

Claude sees 10 individual account stories. It can describe each one, but it can't calculate "25% of Premium accounts activated sofie" because it doesn't know how many Premium accounts there are total — it only has 10 samples.

#### The LLM call (identical for both)

Both pipelines feed their context into the same `JourneyLLM.analyze_with_method()`:

```python
# System prompt (same for both):
"You are an expert B2B SaaS product analyst specializing in
cybersecurity awareness platforms and customer journey analysis..."

# User message (same structure, different context):
"## Account & Product Data Context:
{context from GraphRAG or Naive RAG}

## Product Analysis Question:
Which features have the biggest activation gaps across plan tiers?
Where should the product team focus onboarding improvements?"
```

The only variable is the context block. Claude Sonnet generates a response, same temperature (0.3), same max_tokens (1024).

#### Scoring (heuristic, not human judgment)

Each response is scored by `score_response()` using regex pattern matching:

```python
# Stats cited: counts numbers like "25.0%", "$142,000", "70 accounts"
stats_cited = len(re.findall(r'\d+\.?\d*%|\$[\d,]+|\d+ accounts?|\d+ tickets?', response))

# Features mentioned: counts named features and tiers
features_mentioned = len(re.findall(
    r'report_button|sofie|sso|scim|human_risk_os|elearning|phishing|'
    r'Essential|Professional|Premium|Ultimate', response))

# Actionability: counts recommendation keywords
actionability = len(re.findall(
    r'should|recommend|consider|implement|prioritize|focus|invest|'
    r'improve|reduce|increase|target|automate', response))

total = stats_cited + features_mentioned + actionability
```

**Caveat**: This scoring is crude. It measures *verbosity of data citation*, not *correctness or insight quality*. A response that says "25% of accounts, 30% of accounts, 45% of accounts" three times scores higher than one that says "the key insight is X" once. The side-by-side response text is more informative than the numeric scores.

### What GraphRAG does well (Q1, Q4)

These are **aggregation questions** — they need population-level statistics that can only come from traversing the whole graph.

**Q1 — Feature activation ↔ churn** (GraphRAG: 58, Naive: 32):
GraphRAG's `query_churn_activation()` calls three functions in sequence:
1. `find_churn_paths(G, limit=30)` — walks all churned Account nodes, extracts their full journeys (events, features, tickets)
2. `find_common_patterns(G, {"churned": 1})` — aggregates event sequences across churned accounts into pattern strings with frequency counts
3. `find_activation_gaps(G)` — computes entitled vs activated per feature per tier

The serialized context is ~3,000 chars of structured statistics. Claude then synthesizes:

> *"elearning: Only 3/30 churned accounts activated (10%) — despite being entitled on all plans. basic_phishing: Only 3/30 churned accounts activated (10%)."*

Naive RAG finds 10 similar accounts and describes each one individually — it can't aggregate across all 200.

**Q4 — Activation gaps by tier** (GraphRAG: 45, Naive: 37):
GraphRAG's `query_activation_gaps()` calls `find_activation_gaps(G)` which iterates every Account → Feature edge in the graph and groups by tier. The context is a complete activation rate table (as shown in the example above). Claude can then identify:

> *"SOFIE: Premium 25.0% activation, Ultimate 26.3% activation. These are high-value accounts paying for features they never turned on."*

### Where Naive RAG is competitive (Q3, Q5)

These are questions where **individual account context** is as informative as aggregates, because the document text already contains the relevant signals.

**Q3 — Tier comparison** (Naive: 59, GraphRAG: 41):
The account documents already contain tier, ARR, features, and churn status. When you search for "Essential vs Premium", FAISS surfaces relevant accounts from both tiers. The individual account stories are compelling — specific names, specific ARR, specific unactivated features. Claude weaves them into a comparative narrative with more color than GraphRAG's numeric summary.

**Q5 — Ticket correlation** (Naive: 34, GraphRAG: 25):
Ticket data is included in the account documents ("Tickets (4 total): Billing/Licensing(2), Bug Report(1)"), so naive RAG can surface accounts with billing tickets. GraphRAG's context was more compact (618 chars vs 5,331) — it gave the right insight (billing = churn signal) but with fewer supporting details for Claude to work with.

### Key Takeaway

GraphRAG wins when the question requires **aggregation across the full dataset** — activation rates, pattern frequencies, cohort-level metrics. Naive RAG wins when the question benefits from **individual account stories** with rich context. The ideal system uses both.

At 200 accounts, the gap is narrow because naive RAG's documents are information-dense. At 20K+ accounts (production scale), naive RAG can only surface 10 of 20K, while GraphRAG still traverses the full population — the advantage would widen significantly.

## Honest Assessment

### What works well

- **The data model produces a real insight chain.** You can trace from feature difficulty → low activation → churn → "poor adoption" reason → billing tickets. That's a story Tiago would actually care about, not a synthetic artifact.
- **Conditioning unstructured data on structured context pays off.** The tickets and churn reasons reference specific features and account profiles, so they're coherent with the numbers. Not random filler.
- **The pipeline is reproducible.** `uv sync` + 5 commands, deterministic seeds, everything in one SQLite file. Anyone on the team can regenerate from scratch.

### What's weak

- **GraphRAG only wins 2 of 5 questions.** With 200 accounts, the naive RAG documents are rich enough that similarity search is competitive. The reference repo had 20K sessions — at that scale the gap would be wider. For the demo, this undercuts the "GraphRAG is better" narrative unless we're honest about *when* it wins (aggregation queries) vs when it doesn't matter (individual account lookup).
- **The retrieval functions are shallow.** `find_pre_churn_signals()` counts unactivated features but doesn't actually walk the NEXT edges to find temporal patterns in the 30-60 day window. The graph has temporal chains — we're not using them. That's where the real GraphRAG value would come from.
- **The heuristic scoring is noise.** Regex-counting "stats cited" and "features mentioned" isn't measuring answer quality. A response could cite 20 wrong numbers and score higher than one with 5 correct insights. For the demo, the side-by-side response text is more convincing than the scores.
- **61K events but 60% are admin.login + dashboard_viewed.** That's filler. The interesting events (feature activations, simulation funnel) are < 5% of the data. The graph is big but sparse on signal.

### For the Tiago call

The walkthrough's last section — the 7-point product insight chain — is the strongest demo artifact. It tells a clear story without needing to explain graph theory. Lead with the data observations (Stage 1.3, the churned vs retained activation table) and use GraphRAG vs naive RAG as a "here's how we'd query this at scale" addendum, not the main pitch.
