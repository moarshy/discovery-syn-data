# SynthFlow Synthetic Data Pipeline — Team Walkthrough

This document walks through the full data generation and analysis pipeline for SynthFlow, step by step. Each stage shows what was generated, how to inspect it, and what patterns emerge from the data.

**Pipeline**: Structured data → Unstructured data (Claude) → Graph → Vector index → Evaluation

**To reproduce**: Run from `round2/synthetic-data-gen/`:
```bash
uv sync
export $(grep ANTHROPIC_API_KEY ../../.env | xargs)
uv run python -m src.generate_synthflow
uv run python -m src.generate_unstructured synthflow
uv run python -m src.build_graph synthflow
uv run python -m src.naive_rag synthflow
uv run python -m src.evaluate synthflow
```

**To browse the database**: `uv run datasette data/synthflow.db -p 8788` → open http://localhost:8788

## Stage 1: Structured Data Generation (~10 seconds)

`python -m src.generate_synthflow` creates `data/synthflow.db` with 4 tables.

### How it's generated

No LLM calls here — everything is algorithmic, deterministic (seeded), and fast.

**Config-driven**: A `PLAN_TIERS` dict defines the shape of each tier (Starter / Growth / Enterprise / Agency) with: distribution ratio, ARR range, churn rate, login frequency, campaign frequency, and which features the tier is entitled to. A `FEATURES` list defines 8 features with activation difficulty (low / medium / high). These two configs drive everything downstream.

**Libraries**: Faker for company names and dates, NumPy for distributions (uniform for ARR, lognormal for employee count, weighted random for tier assignment), Python random for coin-flip decisions.

**Key behavioral rules baked into the generator**:
- Churned accounts get **0.5x activation probability** — they were less likely to turn on features
- Churned accounts get **0.6x session count** and **skip ~60% of sessions** in the 30-60 days before their churn date — simulating the activity drop-off before leaving
- Campaign frequency **scales with tier** — Enterprise runs 5x more campaigns than Starter
- Feature activation dates follow an **exponential distribution** (mean 30 days) from contract start — most activations happen early

**State machines for events**: Each account's activity is generated by running state machines that model real platform behavior:
- **Admin session**: `login → dashboard_viewed → [30% chance: agent_created → [70% chance: agent_configured]]`
- **Agent deployment**: `agent.created → agent.configured → agent.deployed(80%) → agent.live(70%) + agent.first_call`
- **Outbound campaign**: `call.initiated(N) → call.connected(60-85%) → call.completed(70-90%) / call.failed + call.transferred(5-15%)`
- **Feature activation**: one event per activated feature, timestamped to the activation date

### 1.1 Accounts (200 rows)

Each row is a B2B customer with plan, revenue, and churn status.

| Plan Tier | Accounts | Avg ARR | Avg Employees | Churn Rate |
|-----------|----------|---------|---------------|------------|
| Starter | 72 | $15,546 | 1,948 | 19% |
| Growth | 57 | $75,783 | 8,558 | 18% |
| Agency | 23 | $211,712 | 26,494 | 13% |
| Enterprise | 48 | $376,915 | 38,366 | 6% |

**Observation**: Churn rate drops from Starter (19%) to Enterprise (6%). Agency is a middle-ground at 13% — these are reseller/multi-tenant accounts that churn when their downstream clients don't see results. Growth's 18% is close to Starter's rate, suggesting the jump from Starter to Growth doesn't meaningfully change retention without feature adoption.


## Stage 2: Unstructured Data Generation (~6 minutes with 10 parallel workers)

`python -m src.generate_unstructured synthflow` adds 4 tables to `synthflow.db`. Each record is generated by Claude, conditioned on the account's structured data (plan, features, churn status).

### How it's generated

Every piece of unstructured data is produced by Claude Sonnet, called via the Anthropic async API with 10 concurrent workers and retry-with-backoff on rate limits. Total: ~70 API calls.

**The key design principle**: each Claude call receives the account's *actual structured data* as context — plan tier, ARR, activated features, unactivated features, churn status. The prompt then tells Claude to generate text that is *consistent with those signals*. This means a churned Starter account with no features activated will get frustrated tickets and negative reviews, while a retained Enterprise account with full activation will get positive reviews and routine QBR transcripts.

**System prompt** (same for all 4 data types):
> *"You are a synthetic data generator for a B2B SaaS AI voice agent platform (similar to SynthFlow). Generate realistic business data that reflects real-world patterns. Return ONLY valid JSON with no markdown fencing or commentary."*

**Account context** (passed to every prompt as JSON):
```json
{
  "account_id": "ACC-0084",
  "name": "Hancock, Peck and Davis",
  "plan_tier": "Starter",
  "industry": "E-commerce",
  "employees": 762,
  "arr": 8900,
  "churned": true,
  "activated_features": ["call_routing"],
  "unactivated_features": ["voice_agents"]
}
```

Below is the gist of each prompt and how it steers Claude's output.

## Stage 3: Graph Construction (~5 seconds)

`python -m src.build_graph synthflow` builds a NetworkX directed graph from all 8 SQLite tables.

### How it's generated

No LLM calls. Reads all 8 SQLite tables and converts them into a NetworkX DiGraph by iterating rows and creating nodes and edges:

1. **Account nodes** (200): one per account, with all profile attributes stored on the node
2. **Feature nodes** (8): one per feature, shared across accounts
3. **Campaign nodes** (3,530): one per campaign
4. **Event nodes** (59,360): one per event
5. **Ticket nodes** (500): one per support ticket
6. **Edges**: `Account→Feature` (ENTITLED_TO/ACTIVATED), `Account→Campaign` (RAN), `Account→Event` (GENERATED), `Campaign→Event` (PRODUCED), `Account→Ticket` (FILED), `Ticket→Feature` (RELATES_TO via a category-to-feature lookup map)
7. **Temporal chains**: Events are grouped by account_id, sorted by timestamp, and linked with NEXT edges — creating a per-account timeline that retrieval functions can walk

### Graph Stats

| Metric | Value |
|--------|-------|
| Total nodes | 63,598 |
| Total edges | 139,719 |

| Node Type | Count | What It Represents |
|-----------|-------|--------------------|
| Account | 200 | Customer organizations |
| Feature | 8 | Product capabilities |
| Campaign | 3,530 | Agent deployments + outbound campaigns |
| Event | 59,360 | Activity events (temporal) |
| Ticket | 500 | Support tickets |

| Edge Type | Count | What It Connects |
|-----------|-------|--------------------|
| GENERATED | 59,360 | Account → Event |
| NEXT | 59,160 | Event → Event (temporal chain) |
| PRODUCED | 15,834 | Campaign → Event |
| RAN | 3,530 | Account → Campaign |
| ACTIVATED | 515 | Account → Feature (turned on) |
| FILED | 500 | Account → Ticket |
| RELATES_TO | 491 | Ticket → Feature |
| ENTITLED_TO | 329 | Account → Feature (has access) |

#### How to explore the graph

```python
from src.build_graph import load_graph, print_graph_stats

G = load_graph("graph/synthflow_graph.pkl")
print_graph_stats(G)

# Inspect one account's neighborhood
acct = "account_ACC-0001"
print(G.nodes[acct])  # Node attributes

# What features is this account entitled to vs activated?
for _, target, data in G.edges(acct, data=True):
    if data["edge_type"] in ("ENTITLED_TO", "ACTIVATED"):
        feat = G.nodes[target]["feature_name"]
        print(f"  {data['edge_type']} → {feat}")

# How many events does this account have?
events = [t for _, t, d in G.edges(acct, data=True) if d["edge_type"] == "GENERATED"]
print(f"  {len(events)} events")

# How many tickets?
tickets = [t for _, t, d in G.edges(acct, data=True) if d["edge_type"] == "FILED"]
print(f"  {len(tickets)} tickets")
```

#### Why the graph matters

The graph enables queries that SQLite joins can't easily answer:
- **Temporal chains**: Follow NEXT edges to see event sequences per account — what happened in what order
- **Multi-hop traversal**: Account → Ticket → Feature (via RELATES_TO) — which features generate the most support friction?
- **Pattern aggregation**: Walk all churned accounts' event chains, extract common patterns, compare to retained accounts
- **Neighborhood comparison**: Compare the full subgraph structure of a churned vs retained account

## Stage 4: Vector Index (~3 seconds)

`python -m src.naive_rag synthflow` creates a FAISS index with 200 documents (one per account).

### How it's generated

No LLM calls. For each of the 200 accounts, `account_to_document()` reads the account's structured data from SQLite (profile, activations, campaigns, events, tickets) and concatenates it into a single text string. These 200 strings are then encoded into 384-dimensional vectors using `sentence-transformers` (all-MiniLM-L6-v2), L2-normalized, and inserted into a FAISS `IndexFlatIP` index for cosine similarity search.

Each document is a text summary of the account:
```
Account ACC-0001 (Smith and Sons) | Plan: Growth | ARR: $48,234 |
Industry: Healthcare | Employees: 3,200 | Churned: False |
Activated features: voice_agents, call_routing, crm_integration |
Entitled but NOT activated: analytics_dashboard |
Campaigns: 12 agent_deployment, 8 outbound_campaign |
Events (380 total): admin.login(120), admin.dashboard_viewed(120), ... |
Tickets (3 total): CRM Integration(2), Agent Configuration(1)
```

This is the **baseline** for comparison. When you ask a question, it finds the 10 most similar account documents by cosine similarity and passes them to Claude. The limitation: it can only surface individual accounts that match the query text — it can't aggregate, compare cohorts, or trace temporal patterns.

## Stage 5: Evaluation (~3 minutes)

`python -m src.evaluate synthflow` runs 5 questions through both GraphRAG and naive RAG, then compares the responses.

### How the evaluation works

Each question goes through two parallel pipelines that differ **only in how they retrieve context** — the LLM, prompt structure, and scoring are identical.

#### Pipeline A: GraphRAG

```
Question
  → Retrieval function walks the NetworkX graph
    (e.g., find_activation_gaps() iterates all 200 Account nodes,
     follows ENTITLED_TO and ACTIVATED edges to Feature nodes,
     groups by plan_tier, computes activation rates)
  → Serialization function converts the graph traversal results
    into a structured text block (markdown tables, bullet lists)
  → Claude receives: system prompt + serialized context + question
  → Response
```

Each of the 5 questions has a dedicated retrieval function in `src/retrieval.py` that knows *which part of the graph to walk*:

| Question | Retrieval function | What it traverses |
|----------|-------------------|-------------------|
| Q1 (churn ↔ activation) | `query_churn_activation()` | All churned Account nodes → their ACTIVATED/ENTITLED_TO edges → Feature nodes. Also runs `find_common_patterns()` to extract event sequences. |
| Q2 (pre-churn signals) | `query_pre_churn_behavior()` | Churned accounts → their GENERATED edges → Event nodes (sorted by timestamp). Counts unactivated features, last event types, ticket categories. |
| Q3 (tier comparison) | `query_tier_comparison()` | All Agency accounts vs all Starter accounts (auto-detected from graph). For each, counts events, campaigns, activated features, tickets, churn rate. |
| Q4 (activation gaps) | `query_activation_gaps()` | Every Account node → every Feature node edge. Groups by tier, computes entitled vs activated counts per feature. |
| Q5 (ticket ↔ churn) | `query_ticket_correlation()` | All Account nodes → FILED edges → Ticket nodes. Splits by churned/retained, counts categories and priorities. |

#### Pipeline B: Naive RAG

```
Question
  → sentence-transformers encodes the question into a 384-dim vector
  → FAISS cosine similarity search finds the 10 closest account documents
  → The 10 document texts are concatenated into a context block
  → Claude receives: system prompt + top-10 documents + question
  → Response
```

#### The LLM call (identical for both)

Both pipelines feed their context into the same `JourneyLLM.analyze_with_method()`:

```python
# System prompt (same for both):
"You are an expert B2B SaaS product analyst specializing in
customer journey analysis and product-led growth..."

# User message (same structure, different context):
"## Account & Product Data Context:
{context from GraphRAG or Naive RAG}

## Product Analysis Question:
Which features have the biggest activation gaps across plan tiers?
Where should the product team focus onboarding improvements?"
```

The only variable is the context block. Claude Sonnet generates a response, same temperature (0.3), same max_tokens (1024).

#### Scoring (heuristic, not human judgment)

Each response is scored by `score_response()` using regex pattern matching:

```python
# Stats cited: counts numbers like "25.0%", "$142,000", "70 accounts"
stats_cited = len(re.findall(r'\d+\.?\d*%|\$[\d,]+|\d+ accounts?|\d+ tickets?', response))

# Features mentioned: counts named features and tiers
features_mentioned = len(re.findall(
    r'voice_agents|call_routing|crm_integration|analytics_dashboard|custom_voices|'
    r'api_access|sla_support|multi_language|Starter|Growth|Enterprise|Agency', response))

# Actionability: counts recommendation keywords
actionability = len(re.findall(
    r'should|recommend|consider|implement|prioritize|focus|invest|'
    r'improve|reduce|increase|target|automate', response))

total = stats_cited + features_mentioned + actionability
```

**Caveat**: This scoring is crude. It measures *verbosity of data citation*, not *correctness or insight quality*. A response that says "25% of accounts, 30% of accounts, 45% of accounts" three times scores higher than one that says "the key insight is X" once. The side-by-side response text is more informative than the numeric scores.

### What GraphRAG does well (Q2)

**Q2 — Pre-churn behavioral signals** (GraphRAG: 27, Naive: 14):
This is an **aggregation question** — it needs population-level patterns across all 30 churned accounts. GraphRAG's `query_pre_churn_behavior()` walks all churned Account nodes, extracts their unactivated features, last event timestamps, and ticket categories. The serialized context is compact (~1,700 chars) but comprehensive — Claude sees that 77% of churned accounts never activated call_routing and 63% never activated voice_agents.

Naive RAG finds 10 individual accounts whose documents mention "churn" — it can describe each one but can't compute "77% of churned accounts" because it only has 10 samples.

### Where Naive RAG is competitive (Q3, Q4)

These are questions where **individual account context** is as informative as aggregates, because the document text already contains the relevant signals.

**Q3 — Tier comparison** (Naive: 57, GraphRAG: 41):
The account documents already contain tier, ARR, features, and churn status. When you search for "Starter vs Enterprise", FAISS surfaces relevant accounts from both tiers. The individual account stories are compelling — specific names, specific ARR, specific unactivated features. Claude weaves them into a comparative narrative with more color than GraphRAG's numeric summary.

**Q4 — Activation gaps** (Naive: 47, GraphRAG: 35):
Naive RAG's documents include activation status per account, so Claude can identify patterns from the 10 most relevant examples. GraphRAG's context was more compact but provided less supporting detail for Claude to work with.

### Key Takeaway

GraphRAG wins when the question requires **aggregation across the full dataset** — pattern frequencies, cohort-level metrics. Naive RAG wins when the question benefits from **individual account stories** with rich context. The ideal system uses both.

At 200 accounts, the gap is narrow because naive RAG's documents are information-dense. At 20K+ accounts (production scale), naive RAG can only surface 10 of 20K, while GraphRAG still traverses the full population — the advantage would widen significantly.

## Honest Assessment

### What works well

- **The data model produces a real insight chain.** You can trace from feature difficulty → low activation → churn → "poor adoption" reason → billing tickets. That's a product story, not a synthetic artifact.
- **Conditioning unstructured data on structured context pays off.** The tickets and churn reasons reference specific features and account profiles, so they're coherent with the numbers. Not random filler.
- **The pipeline is reproducible.** `uv sync` + 5 commands, deterministic seeds, everything in one SQLite file. Anyone on the team can regenerate from scratch.
- **Multi-company support validates the architecture.** The same pipeline (build_graph, retrieval, naive_rag, evaluate, inspect_graph) works for both SoSafe and SynthFlow without code changes — just different data.

### What's weak

- **GraphRAG only wins 1 of 5 questions.** With 200 accounts, the naive RAG documents are rich enough that similarity search is competitive. At 20K+ accounts, the gap would be wider — but for the demo, this undercuts the "GraphRAG is better" narrative unless we're honest about *when* it wins (aggregation queries) vs when it doesn't matter (individual account lookup).
- **The retrieval functions are shallow.** `find_pre_churn_signals()` counts unactivated features but doesn't actually walk the NEXT edges to find temporal patterns in the 30-60 day window. The graph has temporal chains — we're not using them. That's where the real GraphRAG value would come from.
- **The heuristic scoring is noise.** Regex-counting "stats cited" and "features mentioned" isn't measuring answer quality. A response could cite 20 wrong numbers and score higher than one with 5 correct insights. For the demo, the side-by-side response text is more convincing than the scores.
- **59K events but 58% are admin.login + dashboard_viewed.** That's filler. The interesting events (feature activations, agent deployment funnel) are < 5% of the data. The graph is big but sparse on signal.

### SynthFlow vs SoSafe comparison

The two datasets exercise the same pipeline but surface different patterns:
- **SoSafe** has a steeper churn gradient (29% Essential → 5% Premium) while **SynthFlow** has a flatter one (19% Starter → 6% Enterprise with Growth at 18%)
- **SoSafe's** activation gap is most dramatic for IT-heavy features (sso, scim, report_button). **SynthFlow's** gap is most dramatic for the core product (call_routing: 67pt gap) — suggesting voice AI platforms lose customers who never set up the basic telephony
- **SoSafe** churned accounts file 51% billing tickets; **SynthFlow** churned accounts file 44% — same pattern, different magnitude
- The fact that both datasets produce a coherent insight chain from independent generation validates that the pipeline architecture works across domains
